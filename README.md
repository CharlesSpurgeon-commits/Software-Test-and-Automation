# Software-Test-and-Automation
This repository contains selected artifacts from Project One and Project Two in my software testing and quality assurance coursework. These projects demonstrate my ability to design, implement, and test software using automated testing tools and structured development practices.
How can I ensure that my code, program, or software is functional and secure?
I ensure functionality and security by building from clear requirements and validating them with automated testing. In this course, I used unit tests (JUnit) to confirm expected behavior, catch edge cases, and prevent regressions when changes are made. I also focus on defensive programming practices such as input validation, throwing meaningful exceptions when requirements are violated, and avoiding unsafe defaults (for example, preventing null or invalid values from being stored). Security is improved by treating validation rules and error handling as part of the design, limiting what the code accepts, and using tests to confirm the software rejects invalid or unsafe inputs consistently.
How do I interpret user needs and incorporate them into a program?
I interpret user needs by translating requirements into specific rules the program must follow, then implementing those rules in small, testable pieces. I start by identifying what the user is trying to accomplish (the “goal”), then define constraints and acceptance criteria (length limits, required fields, allowed formats, and expected errors). From there, I write tests that represent those requirements before or alongside the implementation. This makes user needs measurable and helps ensure the final program behavior matches what the user expects, including how the program responds when users enter incorrect or incomplete information.
How do I approach designing software?
I approach software design by breaking the problem into components with clear responsibilities, keeping the code modular, and designing for maintainability. For example, separating the data model (like Contact) from the business logic (ContactService) makes the program easier to understand and test. I prioritize clean interfaces, consistent validation rules, and predictable error handling. I also use testing to guide design decisions—if a component is difficult to test, it usually means responsibilities are mixed and should be separated. This approach supports long-term quality because it makes it easier to update features, fix bugs, and add new tests as requirements evolve.
